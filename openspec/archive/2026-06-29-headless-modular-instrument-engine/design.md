## Context

The product is an OSS virtual instrument inspired by Kontakt's instrument-building philosophy, but with routing and scripting placed at the center. The first implementation will be headless so the core engine, patch format, validation rules, and rendering behavior can be tested before any GUI or plugin surface exists.

There is no existing application code in this repository. These artifacts define the initial architecture and acceptance criteria for implementation.

## Goals / Non-Goals

**Goals:**

- Define an engine where instruments are YAML-described modular graphs.
- Support explicit module, port, and cable routing as the core domain model.
- Treat VCA/control routing as first-class and broadly connectable between compatible ports.
- Treat scripts as graph modules that can transform events and control signals.
- Permit feedback only through explicit delay or scheduling boundaries.
- Keep the engine independent of future GUI, standalone, plugin, and library frontends.

**Non-Goals:**

- No GUI, rack view, cable drawing, or patch editor in the initial implementation.
- No VST, CLAP, AU, or realtime audio driver integration in the initial implementation.
- No Kontakt compatibility, proprietary library support, or advanced sample-library features in the initial implementation.
- No unrestricted audio-rate scripting in the initial implementation.

## Decisions

### Decision: YAML patch files

Patch files will use YAML rather than JSON, TOML, or a binary format. YAML is readable for hand-authored patches, supports structured module definitions cleanly, and can later be generated by a GUI without changing the underlying format.

Alternatives considered:

- JSON: easier for tooling, but noisy for hand-authored patches.
- TOML: readable, but less natural for nested module and port declarations.
- Binary format: efficient, but hostile to OSS inspection, diffs, and hand editing.

### Decision: Engine-first architecture

The engine core will not depend on CLI, GUI, or plugin code. Frontends will load patches, provide events/assets, invoke rendering, and consume outputs through engine APIs.

Alternatives considered:

- CLI-first internals: faster to prototype, but risks coupling the engine to command behavior.
- GUI-first internals: closer to the long-term UX, but delays validation of routing semantics.

### Decision: Typed graph with named ports

Modules expose named input and output ports. Connections refer to ports explicitly as `module_id.port_name`. Signal compatibility is validated before processing starts.

Alternatives considered:

- Hardcoded modulation fields: simpler for small patches, but undermines modular routing.
- Untyped cables: flexible, but produces late runtime errors and ambiguous processing behavior.

### Decision: VCA/control is a first-class signal family

VCA/control ports are compatible according to explicit signal typing. Any compatible output can connect to any compatible input, and many-to-one behavior requires explicit mixer/summing modules.

Alternatives considered:

- Special-case VCA destinations inside each module: easier initially, but prevents Reason-style patch cable semantics.
- Treat all control as parameters: convenient for automation, but insufficient for modular-rate routing.

### Decision: Scripts are modules, not host callbacks

Script modules participate in the graph with declared ports, state, and bounded process calls. Scripts can transform MIDI/events/control values and emit outputs, but cannot recursively execute the graph.

Alternatives considered:

- Global patch scripts: close to Kontakt-style scripting, but less modular and harder to reason about.
- Arbitrary DSP scripting: powerful, but too risky for the first realtime-safe architecture.

### Decision: Feedback requires explicit boundaries

Cycles are allowed only when each cycle contains a valid delay or scheduling boundary. Audio feedback requires an audio delay boundary. Control feedback requires a control delay, smoothing, or tick boundary. Event/script feedback is queued to a future tick or block.

Alternatives considered:

- Reject all cycles: simple, but eliminates a core modular-synth behavior.
- Allow all cycles: expressive, but creates algebraic loops, instability, and undefined execution order.

## Risks / Trade-offs

- YAML flexibility can make validation complex -> define a strict schema and fail before rendering.
- Script modules can become a safety and determinism risk -> bound script execution and prohibit recursive graph processing.
- Feedback can create unstable or clipping patches -> validate delay boundaries and provide limiter/saturator modules later as musical safety tools.
- Many-to-one control routing may surprise users if implicit summing is absent -> require explicit mixer modules and make validation errors clear.
- Headless-first development may delay UX validation -> keep patch syntax GUI-ready by using stable module IDs, named ports, and explicit connections.
